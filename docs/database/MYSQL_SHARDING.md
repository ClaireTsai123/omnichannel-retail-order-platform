# MySQL Sharding

## Overview

This project uses Apache ShardingSphere JDBC in `order-service` to split order data across two MySQL databases:

- `order_db_0`
- `order_db_1`

The sharding setup is local-Docker friendly and demonstrates how high-volume order data can be partitioned while the application continues to use one logical datasource.

## Scope

Sharding is currently implemented only for order data managed by `order-service`.

| Logical Table | Physical Tables                                    | Sharding Column | Algorithm           |
|---------------|----------------------------------------------------|-----------------|---------------------|
| `orders`      | `order_db_0.orders`, `order_db_1.orders`           | `id`            | `ds${id % 2}`       |
| `order_items` | `order_db_0.order_items`, `order_db_1.order_items` | `order_id`      | `ds${order_id % 2}` |

Other service databases are not sharded in the current implementation.

## Configuration Files

Key files:

- `order-service/src/main/resources/application.yml`
- `order-service/src/main/resources/sharding.yaml`
- `docker/mysql/init/01-create-databases.sql`
- `docker/mysql/init/02-create-order-shard-tables.sql`

`order-service` uses the ShardingSphere driver:

```yaml
spring:
  datasource:
    driver-class-name: org.apache.shardingsphere.driver.ShardingSphereDriver
    url: jdbc:shardingsphere:classpath:sharding.yaml?placeholder-type=environment
```

`sharding.yaml` defines two physical datasources, `ds0` and `ds1`, mapped to `order_db_0` and `order_db_1`.

## How Routing Works

Order IDs are generated by `order-service` before persistence. ShardingSphere then routes writes and reads based on the configured inline algorithms:

```text
orders      -> ds${id % 2}
order_items -> ds${order_id % 2}
```

This keeps order rows and order item rows aligned by order ID. For example:

```text
orderId = 1001 -> 1001 % 2 = 1 -> order_db_1
orderId = 1002 -> 1002 % 2 = 0 -> order_db_0
```

## Local Docker Initialization

Docker Compose starts a MySQL container and mounts SQL initialization scripts from `docker/mysql/init`.

The initialization scripts create:

- service databases such as `user_db`, `catalog_db`, `inventory_db`, `payment_db`, `ledger_db`, `fulfillment_db`, and `promotion_db`
- order shard databases `order_db_0` and `order_db_1`
- physical `orders` and `order_items` tables in both order shard databases

The MySQL password is provided by environment variable through the local `.env` file. The real `.env` file should remain local and uncommitted.

## Why Sharding Is Useful Here

Order tables are often among the highest-write tables in an ecommerce system. Sharding order data demonstrates:

- horizontal data partitioning
- reduced write pressure on a single database
- transparent logical table access through ShardingSphere
- a foundation for discussing future scaling strategies in interviews

## Current Limitations

This is a local/interview implementation, not a full production sharding platform.

Current limitations:

- only order data is sharded
- no cross-shard distributed transaction manager is configured
- no read replicas are configured
- no online resharding strategy is implemented
- no production migration workflow is implemented
- `sql-show: true` is enabled for visibility during local development

## Production Considerations

A production version would typically add:

- managed database infrastructure
- migration tooling such as Flyway or Liquibase
- shard capacity planning
- read/write splitting or replicas where appropriate
- backup and restore validation per shard
- observability for shard-level latency and error rates
- operational runbooks for resharding and incident response

## Interview Talking Points

- Sharding is scoped to `order-service` because order data is the natural high-growth domain.
- `orders` route by `id`, while `order_items` route by `order_id` so related records stay on the same shard.
- ShardingSphere hides physical database routing from most application code.
- The project demonstrates the pattern locally without requiring managed cloud databases.
